## 1. 目标与范围

### 1.1 核心目标（二阶）

| 编号 | 目标 | 说明 |
|------|------|------|
| G1 | **状态可序列化** | 将任意合法魔方状态编码为固定结构的数据，便于存储、比较、哈希。 |
| G2 | **操作可记录与回放** | 将用户或算法产生的操作序列持久化，支持从初始状态逐步重放至当前状态。 |
| G3 | **两状态间寻路** | 给定源状态与目标状态（或仅目标），计算**最短或近似最短**的基本转动序列（如 U/D/L/R/F/B 及其逆）。 |
| G4 | **可解性判定** | 用户自定义涂色时，判断该配色是否对应**数学上可解**的魔方状态；若可解，再给出解法或提示。 |

### 1.2 非目标（本仓库 / 本文档范围外）

以下内容**不纳入当前项目与本文档正文**，相关材料见 **`目前不考虑做的内容/`** 目录（如三阶建模草案、可选改进列表等）：

- 三阶及更高阶魔方建模与求解
- 其他变体（如 SQ1 等）
- 与二阶无关的通用 N 阶状态编码

本文档内若出现「状态空间较大」等表述，均指**在二阶前提下**的工程取舍（例如 BFS 深度上限、是否预计算全表），不涉及高阶。

### 1.3 适用对象

- **仅二阶（2×2×2）**：与本仓库现有编码（8 元数组 + 位置/朝向，见 `README.md`）直接衔接。

## 2. 术语与符号

| 术语 | 含义 |
|------|------|
| **状态（State）** | 魔方在某一时刻的完整可观测信息（块位置、朝向、或等价的颜色分布）。 |
| **动作（Move）** | 单次合法层旋转，如 `U`、`U'`、`R2`（若支持半转）。 |
| **操作序列（Sequence）** | 有序动作列表 `[m₁, m₂, …, mₙ]`。 |
| **初始状态（S₀）** | 约定好的复原态或项目 README 中的标准配色初始态。 |
| **状态转移** | 对状态 S 施加动作 m 得到 S'，记为 S —m→ S'。 |

## 3. 状态存储

### 3.1 单状态存储

- **用途**：存档当前局面、与服务器同步、作为寻路算法的节点标识。
- **要求**：
  - 与项目内 `get_cube_state()`（或等价接口）输出格式一致或可逆映射；
  - 支持稳定序列化（JSON/二进制）与哈希（用于去重）。

### 3.2 状态链（操作历史）

- **结构建议**：有向链或时间线  
  `S₀ —m₁→ S₁ —m₂→ … —mₙ→ Sₙ`
- **最小存储**：仅存 **动作序列** `[m₁,…,mₙ]` 即可从 S₀ 重算 Sₙ；完整快照可选（便于断点恢复与调试）。
- **元数据**（可选）：时间戳、用户 ID、会话 ID、备注标签。

### 3.3 与「回放」的关系

- **回放**：从 S₀ 依次应用存储的动作序列，驱动 3D 模型与 UI 状态同步更新。
- **校验**：回放终点状态应与当时保存的「单状态快照」一致（若同时保存）。

## 4. 寻路与两状态间操作推导

### 4.1 问题表述

- **输入**：源状态 S_src、目标状态 S_dst（或仅 S_dst，且 S_src = S₀）。
- **输出**：动作序列，使从 S_src 经该序列到达 S_dst；若无解则明确返回不可达/不可解原因。

### 4.2 常见算法路线

| 方法 | 思路 | 优点 | 注意 |
|------|------|------|------|
| **BFS（广度优先）** | 在状态图上从 S_src 扩展，首次到达 S_dst 的层数即最短步数（单步代价为 1 时）。 | 步数最优（在离散动作、无权图假设下）。 | 二阶状态空间有限，可评估全表或带深度上限的 BFS；高阶策略见 `目前不考虑做的内容/`。 |
| **双向 BFS** | 从 S_src 与 S_dst 同时扩展。 | 显著减小程序搜索深度。 | 需规范中间状态相遇判定。 |
| **预计算表 / 模式库** | 对子目标建表再拼接。 | 二阶可考虑全状态距离表（规模可评估）。 | 实现与维护成本随需求上升。 |
| **启发式搜索（A*）** | 定义可采纳的启发函数。 | 可平衡质量与速度。 | 需可证明或可接受的启发设计。 |

### 4.3 文档原述「遍历 + 去短留优」的规范表述

原意可理解为在 **搜索或建图** 过程中：

- 从初始状态 S₀ 出发扩展状态；
- 若某状态 S 已被访问过，且**新到达路径长度不优于**已有记录，则丢弃当前更长或等长冗余分支，仅保留更短（或首次）到达路径。

这在 **BFS 层序扩展** 或 **Dijkstra（边权一致时等价 BFS）** 中通过「已访问集合 + 最短距离」自然实现。若需记录**多条等价最短路径**，则需额外策略（如仅保留一条代表路径）。

### 4.4 「能对任意状态求解」的边界（二阶）

- **数学上**：并非任意涂色都对应可解状态；二阶需满足角块**扭转**与**置换奇偶**等可解性约束（无独立棱块）。
- **产品上**：应区分：
  - **不可解**：明确提示用户调整涂色或检查输入；
  - **可解但搜索超时**：返回近似解或分层提示，而非静默失败。

## 5. 用户涂色与可解性

> **文档归属**：本节的**详细设计**放在本文档（`状态存储.md`），不在主项目 `README.md` 重复展开。  
> 主 `README.md` 仅在「扩展建议 / 文档与功能对照」中**索引**到本节，避免教学主线与算法细节混杂。

### 5.0 与主项目的关系

| 层次 | 内容 | 存放位置 |
|------|------|----------|
| **规范层** | 二阶角块位置编号、朝向编码、初始状态数组格式 | `README.md` |
| **扩展层（本文）** | 用户涂色如何变成状态、是否可解、如何寻路 | `状态存储.md` §5、§4 |
| **实现层** | `Cube2x2.get_cube_state()`、未来 `SolvabilityChecker` 等 | 代码 |

**结论**：「涂色 → 可解性」依赖 README 中的编码定义，但**算法说明、阶段划分、边界条件**以本文为准；实现时建议独立模块（如 `Solvability2x2.gd`），避免塞进 `Cube2x2.gd` 核心旋转逻辑。

### 5.1 从涂色到状态

1. 用户在界面为各贴纸指定颜色（或从相机/导入图案）。
2. 将颜色映射到项目定义的 **块级状态编码**（与 `README.md` 中位置 + 朝向规则一致）。
3. 得到状态 S，进入存储或寻路流程（§4）。

### 5.2 可解性判定（概要，仅二阶）

- 校验角块颜色 multiset、各角三面颜色是否来自标准配色、**扭转与置换**是否满足二阶可解条件（具体判据可在实现阶段单独成表或单元测试）。
- 高阶可解性不在本仓库范围内；若需参考可阅 `目前不考虑做的内容/` 中归档文档。

### 5.3 查表求最短路径（二阶）

- **前提**：可对二阶合法状态空间预计算距离表或运行时 BFS（规模可评估）。
- **流程**：`encode(涂色) → 可解性通过 → 查表或 BFS → 输出动作序列`。  
- **与本项目关系**：作为二阶成熟阶段能力规划；与三阶/分层求解无关内容见 `目前不考虑做的内容/`。

### 5.4 建议实现顺序（与 §6 阶段对应）

1. **先能编码**：涂色 UI → 生成与 `get_cube_state()` 同构的 8 元数组（或等价结构）。  
2. **再判可解**：在数组上跑二阶可解性判据（不依赖 3D 场景）。  
3. **再寻路**：可解时再调用 §4 的 BFS/查表（P1）。  

未通过可解性时，应给用户明确原因类别（如颜色数量不对、奇偶性不满足等——具体以实作时数学表为准）。

## 8. 与本 Godot 项目对接（操作录像 ≡ 批操作）

### 8.1 产品概念统一

| 名称 | 说明 |
|------|------|
| **操作录像** | 按时间顺序记录用户触发的每一步魔方动作，可命名保存，并可逐步或一键回放。 |
| **批操作 / 宏** | 用户将一串动作打包、命名、保存，执行时等价于按序重放。 |

二者在数据模型上**完全一致**：都是 **有序动作列表** + **元数据**（名称、创建时间、可选描述）。区别仅在 UI 入口（「开始录像 / 停止并保存」vs「新建宏并编辑动作列表」）。

### 8.2 现行状态编码能否「直接」用于录像？

| 数据 | 是否适合作为录像主存储 | 说明 |
|------|------------------------|------|
| **`get_cube_state()` 返回的 8 元整型数组** | **不推荐单独作为主存储** | 适合校验、寻路节点、存档「某一帧」；但相邻两帧之间**差分不直观**，且回放需从状态反推动作（二阶上困难且不必要）。 |
| **动作序列**（如 `["U","R","F","U'"]`） | **推荐作为主存储** | 与 `Cube2x2` 已有 `rotate_*` 一一对应；从标准 `S₀` 重放即可复现模型与 UI 状态数组。 |

**结论**：录像/批操作以 **动作序列** 为权威；可选在「停止录像」时额外存一份 `get_cube_state()` 终点快照，用于**自检**（回放后数组应一致）。

### 8.3 当前工程已有接口（无需改魔方核心即可对接）

以下均在 `scripts/Cube2x2.gd`（`class_name Cube2x2`）：

| 接口 | 作用 |
|------|------|
| `rotate_U` / `rotate_D` / `rotate_L` / `rotate_R` / `rotate_F` / `rotate_B` | 各面顺时针一层 |
| `rotate_U_counterclockwise` 等 6 个 | 各面逆时针（对应 `Shift+` 键逻辑） |
| `reset()` | 回到初始构造状态（录像回放前应先调用，除非明确从中间态续播） |
| `get_cube_state() -> Array` | 当前 8 元编码，用于显示与校验 |

输入层 `scripts/UIController.gd` 已把键盘映射到上述 `cube` 方法；**录像应在「与 UI 同一层」或更上层**拦截「即将执行的动作」，避免漏记。

### 8.4 建议新增脚本与职责划分

| 方案 | 说明 |
|------|------|
| **推荐：新建 `OperationRecorder.gd`（或 `CubeMacroManager.gd`）** | 单类负责：`start_recording` / `stop_and_save` / `append_move(move_id)` / `play_macro(name_or_id)` / `load_save_list`。内部只调 `Cube2x2` 的 `rotate_*` 与 `reset`，不复制旋转数学。 |
| **可选：在 `UIController.gd` 内嵌** | 改动集中但文件会变大；适合原型，长期仍建议拆出。 |

**动作 ID 约定**（与存储、回放共用）：使用稳定字符串，例如  
`U, U', D, D', L, L', R, R', F, F', B, B'`  
（与现有 12 种转动一一对应；若日后增加 `R2` 等再扩展枚举。）

### 8.5 回放流程（与批操作相同）

1. `cube.reset()`（若从复原态演示）。  
2. 对序列中每个 `move_id` 调用对应 `cube.rotate_*()`。  
3. 每步后刷新 UI（如 `get_cube_state()` 显示）。  
4. 与 **§9** 一致：若主程序已采用 Tween 转动动画，回放应 **每步 await 动画结束**（或监听完成信号），避免逻辑态超前于视觉。

### 8.6 存储方式：文本 vs 数据库

| 方式 | 适用场景 | 建议 |
|------|----------|------|
| **JSON / 文本文件**（每宏一个文件或一个 `macros.json`） | 条数少、无复杂查询、易版本管理与 Git 备份 | **P0 首选**：实现快、可人工打开检查。 |
| **SQLite（Godot 无内置，需插件或 GDExtension）** | 大量用户、多宏检索、统计 | 中期若宏数量大再上。 |
| **user:// 目录** | Godot 标准可写位置 | 路径用 `user://cube_macros/` 避免污染工程目录。 |

**推荐文件结构示例**（单条宏）：

```json
{
  "schema_version": 1,
  "name": "演示_U_R_逆",
  "created_at": "2026-03-20T12:00:00",
  "initial_reset": true,
  "moves": ["U", "R", "R'", "U'"],
  "optional_end_state": [4, 8, 12, 0, 16, 20, 24, 28]
}
```

## 9. 体验增强与数据层的衔接（转动动画、音效）

> 与 [`README.md`](README.md) **「路线图 — 阶段 A」** 对应。实现顺序上建议 **先 A1/A2，再推进 §6 的 P0**，避免录像回放与瞬时旋转假设纠缠不清。

### 9.1 转动动画（对现有代码的影响）

- **现状**：`Cube2x2.rotate_pieces` 对 pivot **瞬时**旋转 90°，逻辑与表现同步结束。
- **目标**：对 pivot 旋转做 **Tween / AnimationPlayer 插值**（约 0.15–0.25 s），视觉更连贯。
- **对状态存储的含义**：
  - **单步动作**在数据上仍是一条 `move_id`；动画仅拉长「一次状态转移」的墙钟时间。
  - **回放 / 宏**：每步应 `await` 动画结束再执行下一步，或让 `Cube2x2` 在动画完成后发出 `move_finished` 信号，避免连点与状态未提交。
- **建议接口演进**：抽象 `apply_move(move_id) -> void`（同步）或 `apply_move_async(move_id)`（内部 `await` tween），供 `UIController`、未来的 `OperationRecorder` 共用。

### 9.2 音效

- **挂载点**：在「一次层转动开始/结束」触发（与 Tween 的 `started`/`finished` 对齐最佳）。
- **实现**：场景级 `AudioStreamPlayer`（UI 音）或 `AudioStreamPlayer3D`（随魔方位置）；资源放在 `res://audio/`（可后续补充）。
- **与录像关系**：音效**不必**写入宏文件；回放时可选「同步播放」或「用户关闭音效」开关。

### 9.3 与 §6 阶段的推荐顺序

1. **A1 动画** → 稳定 `apply_move` 语义与防重入。  
2. **A2 音效** → 纯表现层。  
3. **§6 P0** 录像/宏 → 依赖明确的「一步完成」边界（动画结束 = 一步完成）。  
4. **§6 P1 / P2** 寻路与涂色 → 与动画正交，仍基于 `get_cube_state()` 与动作模拟。
